## WordPress i18n Functions Reference

| Function | Use case | Output |
|----------|----------|--------|
| `__( 'string', 'domain' )` | Return translated string | `string` |
| `_e( 'string', 'domain' )` | Echo translated string | void (echoes) |
| `esc_html__( 'string', 'domain' )` | Return translated + HTML-escaped | `string` |
| `esc_html_e( 'string', 'domain' )` | Echo translated + HTML-escaped | void (echoes) |
| `esc_attr__( 'string', 'domain' )` | Return translated + attribute-escaped | `string` |
| `esc_attr_e( 'string', 'domain' )` | Echo translated + attribute-escaped | void (echoes) |
| `_n( 'singular', 'plural', $count, 'domain' )` | Singular/plural based on count | `string` |
| `_nx( 'singular', 'plural', $count, 'context', 'domain' )` | Plural with context | `string` |
| `_x( 'string', 'context', 'domain' )` | Context-sensitive string | `string` |
| `_ex( 'string', 'context', 'domain' )` | Echo context-sensitive string | void |
| `esc_html_x( 'string', 'context', 'domain' )` | Context + HTML-escaped | `string` |

### The Golden Rule

**Never nest escaping inside translation functions:**

```php
// ❌ Wrong — double escaping
echo esc_html( __( 'Hello', 'my-theme' ) );

// ✅ Correct — use combined function
echo esc_html__( 'Hello', 'my-theme' );
```

**Never concatenate translatable strings:**

```php
// ❌ Wrong — untranslatable as a phrase
echo __( 'Posted', 'my-theme' ) . ' ' . __( 'on', 'my-theme' ) . ' ' . $date;

// ✅ Correct — translators can reorder words
printf(
    /* translators: %s: post date */
    esc_html__( 'Posted on %s', 'my-theme' ),
    esc_html( $date )
);
```

## Plural Forms

Use `_n()` whenever a count affects whether a string is singular or plural.

### Basic Plural

```php
$count = get_comments_number();

printf(
    /* translators: %d: number of comments */
    esc_html( _n(
        '%d comment',
        '%d comments',
        $count,
        'my-theme'
    ) ),
    number_format_i18n( $count )
);
```

**Use `number_format_i18n()`** — not `number_format()` — for numbers in user-facing output. It respects the locale's decimal separator and thousands separator.

### Plural with Context

```php
$count = count( $projects );

printf(
    esc_html( _nx(
        '%d project',
        '%d projects',
        $count,
        'project count in portfolio section',
        'my-theme'
    ) ),
    number_format_i18n( $count )
);
```

### Languages with Multiple Plural Forms

Some languages (Arabic, Polish, Russian, Slovenian) have more than two plural forms. WordPress handles this correctly — the translator provides the plural formula in the `.po` file header. Always use `_n()` even when your source language only has singular/plural — translators handle the complexity.

## Context-Sensitive Strings

The same English word may translate differently depending on context. Use `_x()`:

```php
// "Post" as verb vs noun
$label = _x( 'Post', 'verb: submit a post', 'my-theme' );
$label = _x( 'Post', 'noun: a blog entry', 'my-theme' );

// "Read more" — context tells translators this is a link label, not instructions
$link_text = esc_html_x( 'Read more', 'link label following a post excerpt', 'my-theme' );

// "Draft" — could mean a noun or adjective
$status = _x( 'Draft', 'post status label', 'my-theme' );
```

### Common Cases Requiring Context

| String | Context needed | Why |
|--------|---------------|-----|
| "Post" | noun vs verb | Ambiguous |
| "Comment" | noun vs verb | Ambiguous |
| "Filter" | noun vs verb | Ambiguous |
| "Back" | direction vs return | Ambiguous |
| "Read more" | link label vs instruction | Length limit may differ |
| Month/day abbreviations | abbreviation | Different rules per language |
| "Reply" | link label vs form heading | Case may differ |
| "Search" | button label vs form placeholder | Case may differ |

## Translators Comments

PHPCS requires a `/* translators: */` comment before any `printf()`/`sprintf()` call that includes a translatable string with placeholders.

### Format

```php
/* translators: %s: author name */
printf( esc_html__( 'Posted by %s', 'my-theme' ), esc_html( get_the_author() ) );

/* translators: 1: opening <a> tag, 2: closing </a> tag */
printf(
    esc_html__( 'Read more about %1$sour story%2$s.', 'my-theme' ),
    '<a href="' . esc_url( get_permalink( $about_page_id ) ) . '">',
    '</a>'
);

/* translators: %d: number of unread notifications */
printf(
    esc_html( _n( '%d notification', '%d notifications', $count, 'my-theme' ) ),
    number_format_i18n( $count )
);
```

### Named Placeholders (for clarity)

When there are many placeholders, use numbered `%1$s`, `%2$s` format so translators can reorder:

```php
/* translators: 1: post title, 2: author name, 3: post date */
printf(
    esc_html__( '"%1$s" by %2$s on %3$s', 'my-theme' ),
    esc_html( get_the_title() ),
    esc_html( get_the_author() ),
    esc_html( get_the_date() )
);
```

## JavaScript Translations

WordPress provides `wp.i18n` for translating strings in JavaScript/block editor contexts.

### Setup in PHP

Register the script, then attach translations:

```php
wp_register_script_module(
    'my-theme-interactions',
    get_template_directory_uri() . '/assets/js/interactions.js',
    array( '@wordpress/interactivity', '@wordpress/i18n' ),
    filemtime( get_template_directory() . '/assets/js/interactions.js' )
);

// Set i18n data for script modules (WP 6.7+)
// For older: wp_set_script_translations( 'handle', 'domain', path )
```

For classic scripts:
```php
wp_enqueue_script( 'my-theme-scripts', ... );
wp_set_script_translations( 'my-theme-scripts', 'my-theme', get_template_directory() . '/languages' );
```

### Using in JavaScript

```js
import { __, _n, _x, sprintf } from '@wordpress/i18n';

// Basic string
const label = __( 'Load More', 'my-theme' );

// Singular/plural
const count = state.posts.length;
const countLabel = sprintf(
    // translators: %d: number of posts
    _n( '%d post', '%d posts', count, 'my-theme' ),
    count
);

// Context-sensitive
const postLabel = _x( 'Post', 'verb: submit a post', 'my-theme' );

// With named placeholders
const message = sprintf(
    // translators: %s: author name
    __( 'Written by %s', 'my-theme' ),
    authorName
);
```

### Generating JSON Translation Files

WordPress needs `.json` files for JS translations (format: `{domain}-{locale}-{script-handle-hash}.json`).

Generate from `.po` files:
```bash
# WP-CLI (recommended)
wp i18n make-json languages/ --no-purge

# Or with wp-scripts
npx wp-scripts i18n make-json languages/
```

The JSON files go in `languages/`. WordPress finds them automatically.

## Block Pattern Translations

Strings inside block pattern PHP files are translatable like regular PHP.

```php
// patterns/hero.php
?>
<!-- wp:heading -->
<h1><?php esc_html_e( 'Your Compelling Headline', 'my-theme' ); ?></h1>
<!-- /wp:heading -->

<!-- wp:paragraph -->
<p><?php esc_html_e( 'Subheading text here.', 'my-theme' ); ?></p>
<!-- /wp:paragraph -->

<!-- wp:button -->
<div class="wp-block-button">
    <a class="wp-block-button__link" href="#">
        <?php esc_html_e( 'Get Started', 'my-theme' ); ?>
    </a>
</div>
<!-- /wp:button -->
```

### Pattern Metadata Translation

The pattern title, description, and category label should also be translation-ready:

```php
<?php
/**
 * Title: <?php esc_html_e( 'Hero with CTA', 'my-theme' ); ?>
 *
 * Note: Block pattern headers are parsed as PHP comments by WordPress.
 * The Title in the header IS translatable when using the register_block_pattern() API,
 * but NOT when using the file-based system (patterns/*.php).
 * For file-based patterns, the title is used as-is from the PHP comment.
 *
 * To make pattern titles translatable, register them programmatically:
 */
```

Programmatic registration with translatable title:
```php
// In inc/block-patterns.php
function my_theme_register_patterns(): void {
    register_block_pattern(
        'my-theme/hero',
        array(
            'title'       => __( 'Hero with CTA', 'my-theme' ),
            'description' => __( 'Full-width hero section with headline and buttons.', 'my-theme' ),
            'categories'  => array( 'hero', 'featured' ),
            'content'     => require get_template_directory() . '/patterns/hero.php',
        )
    );
}
add_action( 'init', 'my_theme_register_patterns' );
```

## theme.json Translations

`theme.json` color palette names, font family names, and spacing size names are translatable via `.pot`:

WordPress does NOT automatically translate `theme.json` values. Names like "Primary" or "Large" appear as-is in the Site Editor color picker. To translate them:

1. Add the `name` values to your `.pot` file manually (or via a custom extraction script)
2. The Site Editor reads from the translated WordPress installation — `theme.json` names currently are NOT passed through `__()` automatically

**For the editor UI:** Site Editor color/font names are English-only in the current WordPress implementation. This is a known limitation. Focus i18n effort on PHP, JS, and pattern strings.

## Generating .pot Files

The `.pot` file is the translation template — all other `.po` / `.mo` files are derived from it.

### Using WP-CLI (recommended)

```bash
# Generate .pot from theme directory
wp i18n make-pot . languages/my-theme.pot \
    --domain=my-theme \
    --exclude=node_modules,vendor,tests \
    --include="*.php,*.js" \
    --package-name="My Theme" \
    --package-version="1.0.0" \
    --file-comment="Translation template for My Theme"

# Verify .pot was created
ls -la languages/my-theme.pot
wc -l languages/my-theme.pot
```

### Using wp-scripts

```bash
npx wp-scripts i18n make-pot . languages/my-theme.pot \
  --domain=my-theme
```

Add to `package.json`:
```json
{
    "scripts": {
        "i18n:pot": "wp i18n make-pot . languages/my-theme.pot --domain=my-theme --exclude=node_modules,vendor",
        "i18n:json": "wp i18n make-json languages/ --no-purge"
    }
}
```

### .pot File Location

```
{{theme-slug}}/
└── languages/
    ├── my-theme.pot          # Template (committed to repo)
    ├── my-theme-fr_FR.po     # French translation (human-edited)
    ├── my-theme-fr_FR.mo     # French compiled (binary, generated)
    ├── my-theme-de_DE.po     # German translation
    ├── my-theme-de_DE.mo     # German compiled
    └── my-theme-fr_FR-{{hash}}.json  # French JS translations
```

Commit `.pot`, `.po`, and `.json` files. Do NOT commit `.mo` files (generated on server).

## Translation Platforms

### GlotPress (WordPress.org)

If the theme is submitted to WordPress.org, translations are managed on translate.wordpress.org. No additional setup needed — the .org team handles it.

### Loco Translate (Self-Hosted)

Popular plugin for self-hosted translation management. Works with existing `.pot` files:

```
1. Install Loco Translate plugin
2. Go to Loco Translate → Themes → {{Theme Name}}
3. Create new language → select locale
4. Translate strings in the browser interface
5. Save — generates .po and .mo files automatically
```

### Transifex

```bash
# Install Transifex CLI
pip install transifex-client

# Initialize project
tx init

# Configure .tx/config
[main]
host = https://www.transifex.com

[my-project.my-theme]
file_filter = languages/<lang>.po
source_file = languages/my-theme.pot
source_lang = en
type = PO

# Push source strings
tx push -s

# Pull translations
tx pull -a

# Compile .mo files
find languages -name "*.po" -exec msgfmt -o "$(basename {} .po).mo" {} \;
```

### Weblate (Self-Hosted)

Open-source alternative to Transifex. Integrates via Git — pushes translated `.po` files to the repository automatically.

## RTL (Right-to-Left) Support

### CSS Logical Properties (Required)

All CSS must use logical properties to work correctly in RTL languages (Arabic, Hebrew, Persian, Urdu):

```css
/* ✅ Logical — works in LTR and RTL */
.card {
    margin-inline-start: var(--wp--preset--spacing--40);
    padding-inline: var(--wp--preset--spacing--50);
    border-inline-start: 4px solid var(--wp--preset--color--primary);
    text-align: start;
    float: inline-start;
}

/* ❌ Physical — breaks in RTL */
.card {
    margin-left: 1.5rem;
    padding-left: 2rem;
    border-left: 4px solid #0052cc;
    text-align: left;
    float: left;
}
```

### HTML `lang` and `dir` Attributes

WordPress sets these automatically based on the site language:

```html
<html lang="ar" dir="rtl"> <!-- Arabic -->
<html lang="he" dir="rtl"> <!-- Hebrew -->
<html lang="en" dir="ltr"> <!-- English -->
```

Do NOT hardcode `lang` or `dir` in templates — always let WordPress set them.

### RTL Stylesheet (Optional)

For complex RTL-specific overrides that logical properties cannot handle:

```php
// In inc/enqueue.php
function my_theme_enqueue_rtl(): void {
    if ( ! is_rtl() ) {
        return;
    }

    wp_enqueue_style(
        'my-theme-rtl',
        get_template_directory_uri() . '/assets/css/rtl.css',
        array( 'my-theme-style' ),
        filemtime( get_template_directory() . '/assets/css/rtl.css' )
    );
}
add_action( 'wp_enqueue_scripts', 'my_theme_enqueue_rtl' );
```

`assets/css/rtl.css` only overrides what logical properties cannot handle (e.g., complex flexbox icon positioning).

### Testing RTL

```bash
# Switch WordPress to Arabic for testing
wp option update WPLANG ar
wp option update blogname "موقعي" --path=/path/to/wordpress

# Or temporarily add to wp-config.php:
define( 'WPLANG', 'ar' );
```

Visual test: flip the site layout in browser DevTools:
```js
document.documentElement.dir = 'rtl';
document.documentElement.lang = 'ar';
```

## i18n Checklist

### PHP Strings

- [ ] All user-facing strings wrapped in `esc_html__()`, `esc_html_e()`, `esc_attr__()`, `esc_attr_e()`
- [ ] No `echo __()` — use `_e()` or `esc_html_e()` instead
- [ ] No `esc_html( __() )` — use `esc_html__()` directly
- [ ] No string concatenation for translatable phrases — use `sprintf()` with `%s` placeholders
- [ ] `_n()` used for all count-dependent strings
- [ ] `_x()` used for context-sensitive strings (Post, Comment, Read more, etc.)
- [ ] `/* translators: */` comments on all `printf()`/`sprintf()` calls with placeholders
- [ ] Text domain matches the theme slug in `style.css` header
- [ ] All strings use the same text domain (no mixing)
- [ ] `load_theme_textdomain()` called in `after_setup_theme`

### JavaScript Strings

- [ ] `@wordpress/i18n` imported in JS files with translatable strings
- [ ] `__()`, `_n()`, `_x()`, `sprintf()` used (not native JS `translate()` or custom)
- [ ] `wp_set_script_translations()` called for scripts with i18n strings
- [ ] JSON translation files generated (`wp i18n make-json`)

### Files

- [ ] `languages/` directory exists in theme root
- [ ] `.pot` file generated and committed
- [ ] `.pot` file name matches text domain (`{{text-domain}}.pot`)
- [ ] `.gitignore` does NOT exclude `.po` or `.json` language files

### RTL

- [ ] No physical CSS properties (`margin-left`, `padding-right`, `border-left`, `text-align: left`, `float: left`)
- [ ] All directional CSS uses logical properties (`margin-inline-start`, `padding-inline`, etc.)
- [ ] RTL layout tested with DevTools `dir="rtl"` override
- [ ] No hardcoded `lang` or `dir` attributes in templates

### WordPress.org Compliance

- [ ] Text domain declared in `style.css` header: `Text Domain: {{text-domain}}`
- [ ] `readme.txt` `Tested up to` is current
- [ ] No untranslated strings visible in admin or frontend
