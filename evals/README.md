# Evals — wp-block-theme-converter

This directory contains the evaluation suite for the wp-block-theme-converter skill. Evals test whether the skill triggers (or correctly avoids triggering) for representative queries, and whether its output meets quality standards when it does trigger.

---

## What Are Evals?

Evals are structured test cases that answer three questions:

1. **Should this query trigger the skill?** (should-trigger)
2. **Should this query NOT trigger the skill?** (should-not-trigger)
3. **Is this query ambiguous enough to warrant a clarifying question?** (ambiguous)

They exist because skill behavior cannot be fully captured in unit tests. The skill's responses are generated by a language model, so correctness must be evaluated against human-readable criteria. Evals make those criteria explicit and repeatable.

---

## Directory Structure

One subdirectory per command:

```
evals/
├── README.md                        # this file
└── convert-to-wp-theme/
    ├── should-trigger.md            # queries that MUST trigger the skill
    ├── should-not-trigger.md        # queries that must NOT trigger the skill
    └── ambiguous.md                 # queries requiring a clarifying question
```

As new commands are added to the skill, add a corresponding subdirectory here with all three eval files.

## How to Run Evals

### Manual (current method)

1. **Install the skill** in a Claude session (Haiku, Sonnet, or Opus — see the model matrix below).
2. **Open `should-trigger.md`** for the command you're testing.
3. **Paste each query** directly into the Claude session (as a user message, not a system prompt).
4. **Evaluate the response** against the "Expected behavior" and "Should NOT do" notes in each case.
5. Repeat for `should-not-trigger.md` and `ambiguous.md`.
6. **Record your results** in the PR description using the format in `.github/PULL_REQUEST_TEMPLATE.md`.

You do not need any tooling beyond Claude and a browser or API client.

### Semi-Automated (future)

A future `scripts/run-evals.mjs` script will use the Claude API to send each query and compare the response against structured pass/fail criteria. Until that exists, manual testing is the standard.

## Pass/Fail Criteria

### should-trigger cases

The eval **passes** when:
- The skill triggers (Claude begins following the skill's command workflow, not a generic response).
- The output matches the "Expected behavior" description in the eval case.
- The output does NOT exhibit any behaviors listed in "Should NOT do".

The eval **fails** when:
- Claude gives a generic answer without following the skill's workflow.
- Claude produces output that violates the skill's quality rules (e.g., inline styles in block markup, hardcoded colors).
- Claude asks an unnecessary clarifying question when enough information was provided.

### should-not-trigger cases

The eval **passes** when:
- Claude answers normally without invoking the WP block theme workflow.
- Claude does not output WordPress theme files, theme.json, or block patterns unprompted.

The eval **fails** when:
- Claude incorrectly switches into theme conversion mode.

### ambiguous cases

The eval **passes** when:
- Claude asks at least one clarifying question before proceeding.
- The clarifying question addresses the specific ambiguity described in the eval case.

The eval **fails** when:
- Claude makes assumptions and proceeds without asking.
- Claude asks an irrelevant or generic clarifying question.

## Adding New Eval Cases

When adding a new command to the skill:

1. Create `evals/<command-slug>/should-trigger.md` with at least 5 queries.
2. Create `evals/<command-slug>/should-not-trigger.md` with at least 3 queries.
3. Create `evals/<command-slug>/ambiguous.md` with at least 3 queries.
4. List which eval cases you ran in your PR description (see the PR template).

When updating an existing command, re-run the full eval suite for that command and include results in the PR.
