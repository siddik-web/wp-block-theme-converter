# Contributing to wp-block-theme-converter

Thank you for improving this skill. Contributions are welcome — bug reports, reference corrections, new examples, and script improvements all help.

---

## Getting Started

1. **Clone the repository**

   ```bash
   git clone https://github.com/siddik-web/wp-block-theme-converter.git
   cd wp-block-theme-converter
   ```

2. **Install Node 20+**

   The validation scripts (`scripts/`) require Node.js 20 or later. No `npm install` needed — all scripts use the Node standard library only.

   ```bash
   node --version  # must be v20.0.0 or higher
   ```

3. **No other dependencies needed** for editing reference files, commands, or templates. PHP and WordPress are only needed if you are testing generated theme output directly.

## Project Structure

```
wp-block-theme-converter/
├── SKILL.md                    # Skill manifest — name, description, trigger phrases, tables
├── commands/                   # One file per slash command (e.g. /convert-to-wp-theme)
├── references/                 # Deep-reference documents read on demand during a task
├── templates/                  # Boilerplate files output verbatim or lightly filled
├── examples/                   # Worked examples and golden reference output
│   └── _generated/             # Golden themes generated by the skill (used by doctor.mjs)
├── scripts/                    # Node.js validation and build scripts (stdlib only)
├── evals/                      # Eval suite: one subdir per command
│   └── convert-to-wp-theme/    # should-trigger.md, should-not-trigger.md, ambiguous.md
└── .github/
    ├── workflows/              # CI workflows
    └── PULL_REQUEST_TEMPLATE.md
```

**Key entry point:** `SKILL.md` is the single source of truth for what commands, references, and templates exist. Any file added to `commands/`, `references/`, or `templates/` must also be listed there.

## Making Changes

### Reference files (`references/`)

- Match the existing heading hierarchy: `##` for top-level sections, `###` for subsections.
- Use pipe tables for structured data (options, mappings, version matrices). Keep tables narrow enough to read without horizontal scrolling.
- Every code block must have a language tag (` ```php `, ` ```json `, ` ```html `, etc.).
- Do not add speculative content — only document what WordPress currently supports.
- When updating version-specific information, state the minimum WP version required in a callout or inline note.

### Command files (`commands/`)

- Match the `Step 1 / Step 2 / Step 3` workflow structure used in existing commands.
- Every command must have a "When to use" section and a "Success criteria" block.
- Commands must reference the relevant reference files they depend on (so Claude reads them at the right time).
- Do not embed large code blocks directly — link to templates or reference files instead.

### Scripts (`scripts/`)

- **Pure Node standard library only.** No `npm install`, no third-party packages, no `require()` of anything outside `node:*` built-ins.
- Every script must pass `node --check scripts/your-script.mjs` before submitting.
- Scripts must print clear PASS/FAIL output with a summary exit code (0 = all pass, 1 = any fail).
- Use ES module syntax (`import`/`export`). File extension must be `.mjs`.
- Keep scripts focused: one script, one job.

### Templates (`templates/`)

- Keep templates minimal. No speculative features — only what a typical project of that type needs.
- Use `{{placeholder}}` syntax (double curly braces) for values the skill fills in at generation time.
- Every template must be valid in its target language without substitution (use sensible placeholder defaults).
- PHP templates must follow WordPress Coding Standards (tabs, `snake_case`, `esc_*` output).

## Testing Your Changes

### 1. Validate the skill structure

Run this after any change to `SKILL.md`, `commands/`, `references/`, or `templates/`:

```bash
node scripts/validate-skill.mjs
```

This checks that every file listed in `SKILL.md` exists, that no internal links are dead, and that required meta-files (`CHANGELOG.md`, `LICENSE`, `CONTRIBUTING.md`) are present.

### 2. Test scripts against the golden theme

Run this after any change to `scripts/`:

```bash
node scripts/doctor.mjs examples/_generated/landing-page-theme
```

The golden theme in `examples/_generated/landing-page-theme/` is the reference output the doctor scripts are calibrated against. All checks should pass on it.

### 3. Run evals

After changing a command or reference file, run the affected eval cases manually:

1. Install the skill in a Claude session.
2. Paste each query from `evals/<command>/should-trigger.md` and verify the skill triggers and produces output matching the expected behavior.
3. Paste each query from `evals/<command>/should-not-trigger.md` and verify the skill does NOT trigger.
4. Paste each query from `evals/<command>/ambiguous.md` and verify Claude asks a clarifying question.

Document your eval results in the PR description (see the PR template).

For CI-automated eval guidance, see `evals/README.md`.

## PR Requirements

- **Author ≠ reviewer.** Do not merge your own PR.
- **Eval results required.** The PR description must include the output of running affected eval cases. The PR template has a checklist — fill it in.
- **Link the relevant command or reference.** If your PR touches `commands/wp-pattern.md`, say so in the summary.
- **One logical change per PR.** Don't bundle a reference update and a new command in the same PR unless they are tightly coupled.
- **`node scripts/validate-skill.mjs` must pass.** CI enforces this, but run it locally first.

---

## Code of Conduct

- Be constructive. Critique the change, not the person.
- When reviewing PHP examples, follow [WordPress Coding Standards](https://developer.wordpress.org/coding-standards/wordpress-coding-standards/php/). Flag deviations — don't silently rewrite to your personal style.
- When reviewing block markup or theme.json, refer to the WordPress Block Editor Handbook and the FSE specification. Opinion-only objections ("I would have done it differently") are not grounds for blocking a PR.
- Assume good faith. If something looks wrong, ask before rejecting.
